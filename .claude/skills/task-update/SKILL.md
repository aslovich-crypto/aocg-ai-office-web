# Ведение трекера задач (task-update)

Источник истины по задачам — `docs/TASKS.md` (`AOCG-TASKS-001`). При изменении статуса задача правится в этом файле, и правка коммитится вместе с работой по задаче.

## Когда применять
Команды вида: «закрой E1», «начни S-07», «#6 в работу», «заблокируй FIN», «помечай задачу <ID> как …», «обнови трекер», а также создание/починка самого `TASKS.md`.

## Легенда (соблюдать строго)
Приоритет: 🔴 высокий · 🟡 средний · 🟢 низкий.
Статус: ✅ готово · 🔵 в работе · ⬜ к работе · ⏸ ждёт/заблокировано · 👀 наблюдаем.

## Формат файла (соблюдать и восстанавливать)
Файл устроен так, и этот формат нужно поддерживать при любой правке:

1. **Шапка**: заголовок, строка `**Документ:** … · **Обновлён:** ДД.ММ.ГГГГ` (дата последней правки — держать актуальной, см. шаги), и легенда приоритетов и статусов.
2. **Блоки по доменам** — каждый блок это заголовок `##` + markdown-таблица. Порядок блоков: До онбординга → Организация → Продукт → Безопасность → Категории → Позже/R&D → Финансы → Хвосты.
3. **Колонки таблицы** строго: `| ID | Задача | Приоритет | Статус | Примечание |`. (Блок «Хвосты» допускает упрощённые колонки `| ID | Задача | Статус | Примечание |`.)
4. **Доменные префиксы ID**: `S-` безопасность, `Cat-` категории, `E-` организация, `T` хвосты, числа — продукт.
5. **Секция «✅ Закрыто недавно»** внизу — короткой строкой через `·`.

При правке статуса менять только нужные ячейки. Не ломать таблицы, не переставлять блоки, не менять колонки. Если файл утёк (таблица побилась) — выровнять под этот формат.

## Шаги при смене статуса
1. Найти задачу по ID (поиск по столбцу ID, не по тексту).
2. ID не найден — не создавать строку молча; сообщить и спросить.
3. Сменить символ в столбце «Статус» по легенде.
4. Обновить «Примечание» коротко (что сделано / чего ждут).
5. Проверка по КОДУ (источник истины статуса — код, не память чата):
   - переход в ✅ — НАЙТИ в коде сигнал реализации (таблица БД, роутер/эндпоинт, компонент, функция) ИЛИ коммит этой сессии. Найдено однозначно → отмечай ✅ сам. Реализация ЧАСТИЧНАЯ (например, бэкенд есть, UI нет) → отметь с пометкой «частично: …» и спроси, закрывать ли. Кода/артефакта нет → не закрывать, предупредить.
   - переход в 🔵 — допустимо без артефакта; но НЕ ставить 🔵, если в коде лишь заглушка-статус без реализации — это ⬜.
6. Обновить поле **«Обновлён»** в шапке на ТЕКУЩУЮ дату сессии (формат ДД.ММ.ГГГГ). Делать при ЛЮБОЙ правке файла, не только смене статуса — дата не должна отставать.
7. Закоммитить `docs/TASKS.md` ВМЕСТЕ с коммитом самой задачи: один коммит = закрытая задача + её отметка. Отдельный коммит только ради статуса — лишь когда работы по коду нет (например, внешний блокер снят).

## Сверка статуса с кодом (стандартное поведение)
Статус в трекере отражает РЕАЛЬНЫЙ код, а не память чата. Пользователь не должен сверять вручную.
- **При закрытии/смене статуса** — проверить именно эту задачу в коде (грепом найти сигнал: имя таблицы, роутер, эндпоинт, компонент). Очевидное отмечать самому; неоднозначное (частично / бэкенд без UI / заглушка) — с пометкой и вопросом.
- **Полная сверка** (все задачи против кода) — по команде вида «сверь трекер», «что сделано по коду», «проверь задачи». Пройти по открытым задачам, найти/не найти реализацию, предложить правки статусов одним списком. Сверять оба репозитория (бэкенд + фронт), если задача их касается.
- Авторитет правок: очевидное — менять сам; спорное — предлагать и ждать «ок». НИКОГДА не ставить ✅ без кода/артефакта.

## Автопечать списка после пуша
После того как фича из трекера ЗАКОММИЧЕНА и её пуш ПОДТВЕРЖДЁН (пользователь сказал «запушил/пуш прошёл» ИЛИ видно, что ветка синхронизирована с origin — `git log origin/main..main` пуст) — автоматически вывести полный список задач (таблицами по блокам, см. формат). На одном коммите БЕЗ подтверждённого пуша — НЕ печатать. Цель: после каждой отгруженной фичи пользователь сразу видит актуальную картину, ничего не запрашивая.

## Связь с журналом сессий
Если ведётся журнал сессий — при закрытии задачи добавить туда строку со ссылкой на ID (что закрыто, каким коммитом). Связь трекер ↔ журнал держится через ID; содержимое не дублируется: трекер — состояние, журнал — история.

## Чего не делать
- Не пересобирать список задач заново — править существующий файл.
- Не менять приоритет, формат таблиц или структуру блоков без явной просьбы.
- Не ставить ✅ раньше, чем работа реально зафиксирована.
- Не вести статусы в чате вместо файла — источник истины только `docs/TASKS.md`.

## Правило коммита (закреплено)
Закрытие задачи и отметка в трекере — один коммит. По истории git должно читаться, какой коммит какую задачу закрыл.
