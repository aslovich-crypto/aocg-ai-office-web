# AOCG AI Офис — фронтенд · инструкции для AI-агента

Постоянные правила для AI-агента (Claude Code). Соблюдать во всех сессиях. Это «оперативная память» репозитория — обновляется вместе с кодом.

## Что это за проект
Фронтенд платформы AOCG AI Офис — B2B SaaS для российского малого и среднего бизнеса. Управление первичными финансовыми документами и управленческим учётом. Мобильно-ориентированный интерфейс по образцу банковских приложений (Т-Банк, Альфа-Банк).

## Технологический стек
- React + Vite
- Иконки: lucide-react
- Графики: recharts
- QR: jsqr
- Хостинг: Railway (auto-deploy из main)

## Команды
- Запуск dev: `npm run dev` (localhost:5173, hot-reload)
- Сборка: `npm run build`

## Ключевые файлы
- `src/App.jsx` — весь UI, компоненты, роутинг (state-based, без URL-роутера). Это монолит, унаследованный технический долг — НЕ дописывать в него новое (см. раздел структуры ниже).
- `index.html` — Inter подключён, viewport-fit=cover
- `vite.config.js` — allowedHosts настроены

## Структура репозитория и лимиты на размер
Текущее состояние: весь интерфейс в одном `src/App.jsx` (несколько тысяч строк). Это долг, а не образец. Цель — не наращивать монолит, а постепенно разносить.

Куда класть новый код (для нового и при касании старого):
- Новый экран/раздел — отдельный файл в `src/pages/` (или `src/screens/`), НЕ дописывать в `App.jsx`.
- Переиспользуемый компонент (кнопка, sheet, баннер) — отдельный файл в `src/components/`.
- Логика запросов к API — в `src/lib/` (или `src/api/`), не размазывать fetch по компонентам.
- Кастомные хуки — `src/hooks/`.
- Статьи помощи — `src/help/` (md через react-markdown).

Правило при работе с App.jsx:
- Если правка затрагивает кусок `App.jsx` — по возможности вынести этот кусок в отдельный файл, а не редактировать на месте и не добавлять рядом. Дробим монолит по ходу дела, а не отдельным «большим рефакторингом».
- Новый функционал в `App.jsx` не добавлять в принципе.

Лимиты (сигналы пересмотреть, не жёсткие запреты):
- Компонент > ~200 строк → сигнал выделить под-компоненты или хук.
- Дублирование разметки/логики в 2+ местах → вынести в общий компонент.
- При приближении к лимиту — сообщить и предложить, как разнести, а не молчать.

## Терминология (строго)
- Платформа — AOCG AI Офис целиком
- Приложение — Прима, Финансы
- Раздел — внутри Примы (Чеки, Авансовые отчёты, Акты, Счета)
- В шапке справа от логотипа — название текущего РАЗДЕЛА (напр. «Чеки»), не приложения. «Прима» в UI не показывается.

## Паттерны «делай»
- Использовать готовые библиотеки (lucide-react, recharts), не самописное.
- Segmented control для периода, bottom-sheet (выезжающая снизу панель) для выбора сотрудника — не dropdown.
- React автоматически экранирует `{переменная}` — это базовая защита от XSS.
- Атомарные коммиты, Conventional Commits.

## Паттерны «не делай»
- НЕ коммитить секреты — только в Railway → Variables. `.env` в `.gitignore`.
- НЕ использовать `dangerouslySetInnerHTML` без явной санитизации.
- НЕ копировать стиль Stripe / Linear / Revolut — сохранять АОЦГ-идентичность.
- НЕ навязывать подсказки (см. UX-принципы ниже).

## UX-принципы подсказок
- Подсказка = приглашение, не лекция. Появляется по запросу, не навязывается.
- Иконки `?` молчат до тапа, нейтральный серый, не отключаемые.
- Янтарные предупреждения — только при реальной проблеме, без мигания, отключаемые в Настройках.
- Onboarding — только при первом действии, с кнопкой «Не показывать снова», 2–3 шага максимум.
- Запрещено: подсказки при каждом действии, бейджи «новое!» на месяцы, неотключаемое обучение.

## Рабочий процесс
- Одна фича — одна сессия, один атомарный коммит.
- Сначала план (plan mode), код не писать до одобрения.
- Показывать diff до применения; коммит — отдельной командой после проверки в браузере.
- UX-решения фиксировать ДО написания кода.
- После фронт-коммита — проверить, что новый бандл доехал до прода (по строковому маркеру, не по хешу бандла).

## Формат коммитов
`тип(scope): краткое описание`
Типы: feat, fix, chore, docs, refactor, test.
Пример: `feat(receipts): уплотнение фильтров, удаление бейджей`

## Чего НЕ касаться без явного запроса
- Конфигурацию Vite и деплоя.
- Общие компоненты шапки при правке отдельного экрана.
- Логику запросов к API без обсуждения.
