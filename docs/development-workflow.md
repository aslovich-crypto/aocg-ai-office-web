# DEV-FLOW — Цикл AI-ассистированной разработки

Универсальный регламент работы над проектом с участием AI-кодинг-агента (Claude Code, Cursor, аналоги). Не привязан к конкретному стеку, хостингу или IDE. Опирается на индустриальные практики 2025–2026: малые партии изменений, планирование до кода, атомарные коммиты, верификация, точки отката.

---

## 01. Базовый принцип — одна фича, одна сессия

Единица работы — **одна небольшая проверяемая фича**, а не крупная цель.

Плохо: «доделать раздел заказов».
Хорошо: «уплотнить блок фильтров на странице заказов на 25% по вертикали».

Закрыли фичу → коммит → следующая. Цель: каждый коммит оставляет рабочую версию проекта.

**Почему это работает.** AI-агент ведёт себя как сильный, но буквальный junior-to-middle инженер: отлично выполняет хорошо определённые задачи и ошибается на размытом или раздутом контексте. Узкая задача = меньше точек принятия решений = выше шанс пройти их все без ошибок.

---

## 02. Планирование до кода — не перестраховка, а математика

Если агент принимает верное решение с вероятностью 80%, а фича содержит 20 точек принятия решений, вероятность пройти все безошибочно ≈ 0.8²⁰ ≈ 1%.

Планирование сворачивает 20 неоднозначных решений в одну проверенную спецификацию, где каждое решение уже принято человеком. Поэтому:

1. Сначала обсудить и зафиксировать подход (в отдельном канале для размышлений, не в коде).
2. Использовать **plan mode** агента, если он есть.
3. Давать фразу-предохранитель: «составь план, не пиши код пока я не одобрю». Без неё агент часто начинает писать сразу, пропуская ревизию.

---

## 03. Цикл одной фичи

**Шаг 1. Постановка — канал размышлений.**
Сформулировать: что меняем, где, как должно стать после. Получить готовый промпт/спецификацию (структура — см. DEV-PROMPT).

**Шаг 2. План — агент.**
Отдать задачу агенту в plan mode. Агент возвращает план без правок кода. Просмотреть, одобрить или скорректировать.

**Шаг 3. Применение — агент.**
Агент применяет правки и показывает diff. Просмотреть изменения. Не доверять «выглядит правильно» — читать diff.

**Шаг 4. Проверка — руками.**
Запустить приложение, пройти сценарий вручную. Чем меньше вы способны проверить вывод сами, тем большего внимания он требует: уверенный, правдоподобный код бывает тонко неверным.

**Шаг 5. Коммит и деплой.**
Коммит атомарный, сообщение в одну строку про одну фичу (формат — см. §06). Деплой по процессу проекта.

**Шаг 6. Фиксация.**
Одной строкой записать сделанное в журнал сессий проекта.

## 04. Разделение каналов

| Что | Где |
|---|---|
| Код и правки | AI-агент в IDE |
| Архитектура, спорные решения, выбор подхода | Канал размышлений (чат) |
| Постоянный контекст проекта (правила, конвенции) | Файл `CLAUDE.md` в репозитории |
| Методология, тон работы с агентом | Отдельный методологический документ |
| История сессий | Журнал сессий |

**Принцип:** каналы не смешиваются на лету. Возник архитектурный вопрос в середине сессии кода — выйти, обсудить в канале размышлений, вернуться с уточнённой задачей.

### Деление канала размышлений: методологический контур и профильные русла

Канал размышлений (чат с агентом для обсуждения, не для кода) делится на параллельные русла, которые стоит вести в **разных чатах**, иначе они смешиваются. Русел два типа.

- **Методологический контур** (один) — предмет обсуждения сам процесс и инфраструктура: регламенты, шаблоны, структура репозиториев, документы, принципы, скиллы. Результат применяется ко многим будущим задачам и переживает их.
- **Профильные русла** (несколько, по доменам проекта) — каждое ведёт свою тему: конкретное приложение/модуль, дизайн-система, платежи, юридические документы, безопасность и т.д. Внутри профиля обсуждают конкретные задачи этого домена: аудит под задачу, решение по ней, промпт для агента, проверка статуса. Результат живёт до закрытия задачи.

Сколько профильных русел и как они называются — зависит от проекта. Маленький проект может обойтись одним руслом «разработка»; большой разносит домены по отдельным чатам. Конкретная карта русел проекта (какие чаты есть и что в каждом) — это проектная специфика, её место в проектном документе, не в этой универсальной методичке.

**Тест границы:** «Если текущая задача завтра закроется, мой вопрос всё ещё актуален?» Да → методологический контур. Нет → профильное русло (то, чьего домена касается задача).

**Тонкость, которая чаще всего сбивает.** Разница не в жанре материала, а в сроке его жизни. Универсальный *шаблон* промпта — методология: он переживёт любую задачу. *Промпт под конкретную задачу* — профильное русло: задачу закрыли, промпт больше не нужен. Один и тот же жанр, разные чаты.

**Сигнал дрейфа.** Если в методологическом контуре начинается промтинг конкретной задачи или проверка её статуса — это сигнал остановиться и открыть нужное профильное русло. И наоборот: если в профильном русле обсуждение уходит в общий процесс — это материал для методологического контура.

## 05. Постоянный контекст — файл `CLAUDE.md`

Главный недоиспользуемый рычаг. Файл в корне репозитория с инструкциями под проект, которые агент подхватывает автоматически в каждой сессии. Туда выносится всё, что иначе пришлось бы копировать в каждый промпт:

- правила дизайн-системы (цвета, шрифты, типографика);
- конвенции кода и именования;
- паттерны «делай / не делай»;
- ссылки на ключевые файлы и точки входа;
- команды сборки, тестов, запуска;
- что агенту запрещено трогать.

Шаблон — см. отдельный документ CLAUDE-TEMPLATE.

Эффект: промпты становятся короче, агент перестаёт нарушать стиль и конвенции, контекст не теряется между сессиями.

---

## 06. Атомарные коммиты и сообщения

Частые чистые коммиты — то, что позволяет откатиться, если агент ошибся в середине сессии, не распутывая кучу смешанных изменений.

Формат сообщения (Conventional Commits): `тип(scope): краткое описание`.

Типы: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`.

Примеры:
- `feat(orders): уплотнение фильтров, удаление номерных бейджей`
- `fix(auth): блокировка после 5 неверных попыток`
- `refactor(api): параметризация SQL-запросов`

Один коммит = одно логическое изменение. Не смешивать фичу и рефакторинг в одном коммите.

---

## 07. Точки отката — обязательны

Перед правками в рабочей системе агент должен показать, как откатить изменения:
- код — `git revert <хэш>`;
- миграция БД — обратный DDL (DROP COLUMN / DROP INDEX / DROP TABLE);
- конфигурация — прежние значения переменных.

Без явной точки отката не начинать правки в продакшене.

---

## 08. Верификация и тесты

- Дать агенту способ проверить себя: тесты, линтер, запуск, curl-маркер на проде.
- После любой правки фронта — убедиться, что новый бандл реально доехал до прода (проверка по строковому маркеру, не по хешу бандла).
- При смене продакшен-логики — зеркально обновлять тестовые моки, иначе тесты ложноположительны.
- Длинная сессия не должна быть единственным носителем решений: коммитить часто, выгружать прогресс в файлы, относиться к каждой сессии как к одноразовой.

---

## 09. Skills — для повторяющихся задач

Любой рабочий процесс, который повторяется чаще раза в неделю с предсказуемым результатом, стоит оформить как Skill: научить агента один раз, довести вывод до стандарта, сохранить — дальше применяется автоматически без повторных объяснений.

Кандидаты: шаблоны промптов под типовые задачи, паттерны код-ревью, регламент аудита перед миграцией, формат документации.

---

## 10. Защитные правила

1. Перед началом фичи рабочая ветка чистая, последний коммит работает.
2. Никогда не две задачи в одной сессии.
3. Правка задела фронт и бэк — деплоят и проверяют оба.
4. Сомнения в подходе — сначала канал размышлений, потом агент.
5. Коммит только после ручной проверки.
6. В промпте/режиме всегда «сначала план, не реализуй до одобрения».

---

## 11. Признаки здорового цикла

- Сессия длится 15–40 минут, не часы.
- Сообщение коммита умещается в строку и описывает одну фичу.
- После коммита можно показать конкретное изменение в UI или API.
- Журнал сессий пополняется на 1–3 записи за день работы.
- Не приходится возвращаться к «почти закрытым» задачам через неделю.

## 12. Управление задачами

То, что превращает «список дел» в управляемый поток: как нарезать задачи, как вести реестр, как учитывать время.

### Сегментирование
- Одна задача = один наблюдаемый результат, закрываемый за одну сессию (см. §01). Не «сделать отчёты», а «детальный режим отчёта: фильтр по сотруднику».
- Признаки хорошо нарезанной задачи: есть критерий готовности, не зависит от незакрытых соседей, помещается в одну сессию.
- Тип задачи помечать заранее: шаблонная / переиспользование известного паттерна / новый тип. Тип влияет на оценку времени (см. ниже).

### Реестр задач
- Единый список: ID, краткое описание, домен/профиль, приоритет, статус.
- Приоритеты тремя уровнями (например 🔴 блокер / 🟡 важное / 🟢 потом).
- Сегментировать по доменам через префикс ID (например `S-` для безопасности), чтобы очередь профиля читалась отдельно. Это перекликается с профильными руслами из §04.
- Один источник истины статусов: задача живёт в реестре, другие документы ссылаются, а не дублируют — иначе статусы расходятся (та же логика, что для карт в §13).
- Закрытие фиксируется: коммит + запись в журнал/реестр.

### Учёт времени
- Цель — калибровка будущих оценок, не давление и не отчётность.
- Измерять активное рабочее время: паузы дольше нескольких минут вычитать.
- Оценка ДО задачи: найти ближайший аналог в накопленной статистике, взять его коэффициент. Для нового типа задачи — явно «оценки нет до первого факта», не выдумывать число.
- После задачи: записать факт, пополнить статистику. Со временем коэффициенты по типам стабилизируются.
- Наблюдение, подтверждённое практикой: детальный аудит перед задачей резко ускоряет исполнение — агент идёт по готовому списку, а не ищет. Аудит окупается во времени, а не тратит его.

Конкретная реализация — таблица коэффициентов по типам, формат журнала времени, шкала приоритетов — это проектная специфика и живёт в проектных документах, не в этой универсальной методичке.

## 13. Семейство карт проекта

Карты (диаграммы) проекта строятся по единому паттерну `docs/*-map.md`: источник-истина + Mermaid под ним. Но уровень автоподдержки зависит от того, какой у карты источник.

- **Полностью живая.** Источник — машинно-читаемая таблица, часто меняется (например статусы задач). Скилл читает таблицу, перерисовывает Mermaid, проверяет покрытие. Пример: карта безопасности из реестра задач.
- **Полу-живая.** Источник частично в коде (структура модулей, таблицы БД, интеграции). Скилл не рисует с нуля, а *сверяет* диаграмму с фактической структурой и сигналит расхождения — ловит дрейф, когда код ушёл вперёд, а карта отстала. Пример: карта архитектуры.
- **Стабильная.** Источник — процесс, меняется редко, не машинный. Mermaid правится вручную прямо в документе, скилл не нужен. Пример: карта методологии (живёт как Mermaid в этом файле).

**Принцип:** живой скилл-генератор оправдан только там, где источник машинно-читаемый и часто меняется. Где стабильно или не машинно — Mermaid правится вручную. Не плодить скиллы ради скиллов.

**Стандарт оформления:** Mermaid (не SVG) для всех живущих карт — текст, виден git-diff, рендерится в GitHub и VS Code. SVG — только для разовых «парадных» схем. Поверх всех карт — единый визуальный стиль (палитра, сетка), который можно закрепить отдельным скиллом-стилем, когда карт станет несколько.

## 14. Источники подхода

Практики основаны на материалах по работе с AI-кодинг-агентами 2025–2026 (Anthropic Claude Code best practices, инженерные разборы рабочих сессий) и устоявшихся методологиях: trunk-based development, Conventional Commits, малые партии изменений (DORA / Accelerate), code review, CI/CD.
